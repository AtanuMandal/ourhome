# Login and Authentication

## Overview
The login module handles user authentication, multi-tenancy selection, and session management. It supports two login methods: **OTP-based login** (primary) and **password-based login**. Multi-society and multi-apartment users are handled with a selection step after credential verification.

---

## Login Methods

### Method 1: OTP Login (Primary)
1. User enters their **email address**.
2. System sends a **6-digit OTP** via SMS to the registered phone number.
3. User enters the OTP within the expiry window.
4. On success, a **JWT token** is issued and returned with the user's profile.

### Method 2: Password Login
1. User enters **email** and **password**.
2. If credentials are valid, a JWT token is issued.
3. If the user belongs to multiple societies, a **selection step** is returned (see Multi-Society Handling below).

---

## Multi-Society and Multi-Apartment Handling

A single email can be registered in:
- Multiple societies (with any role)
- Multiple apartments within the same society (e.g., owns two flats)
- Multiple roles within the same society

### Selection rules:
- If the user belongs to **only one society, one apartment, and one role** → token is issued immediately with no selection step.
- If the user belongs to **multiple societies** → the response returns a list of `LoginOptionDto` entries for the user to select from. Login proceeds after selection.
- If the user is `SUAdmin` → **no apartment selection** is required (admins are not linked to a specific apartment).
- If `SUUser` belongs to **multiple apartments** within one society → apartment must be selected.

### LoginOptionDto fields:
- `UserId`, `SocietyId`, `SocietyName`, `ApartmentId` (null for admins), `ApartmentLabel`, `Role`, `ResidentType`

---

## JWT Token

The JWT is signed with **HS256** and contains:

| Claim | Description |
|-------|-------------|
| `sub` | User ID |
| `email` | User email |
| `role` | User role (`SUAdmin`, `SUUser`, `SUSecurity`, `HQAdmin`, `HQUser`) |
| `societyId` | Society the session is scoped to |
| `apartmentId` | Primary apartment ID (only for `SUUser`) |
| `jti` | Unique token identifier |
| Expiry | Configurable via `JwtExpiryHours` in settings |

- Token expiry causes automatic logout on the frontend (proactive timer + on-load check).
- No refresh token mechanism currently — users must re-authenticate on expiry.

---

## Password Reset

1. `POST /api/auth/password-reset/request` — user provides email; system sends an OTP.
2. If the email is registered in multiple societies, a `userId` must be selected first.
3. `POST /api/auth/password-reset/confirm` — user provides `userId`, OTP, and new password.
4. On success, the password is updated.

---

## Self-Registration

- Users register via a society-specific encrypted invite link generated by `SUAdmin`.
- Registration form requires: full name, email, phone, password, and **confirm password** (both required).
- After registration, the user is redirected to the **login page** — never to the password reset page.
- The user must then log in, select their apartment, and wait for `SUAdmin` approval.

---

## Security Rules

- JWT tokens are verified on every authenticated API request.
- `societyId` in the JWT is validated against the `societyId` in the route on every request.
- Passwords are stored as BCrypt hashes; raw passwords are never stored.
- OTPs expire after a configurable period and are single-use.
- Invite tokens are short-lived JWTs with a `type=invite` claim.

## API Endpoints

| Method | Route | Auth | Description |
|--------|-------|------|-------------|
| `POST` | `/api/auth/login` | Public | Password-based login |
| `POST` | `/api/societies/{id}/auth/request-otp` | Public | Request OTP for phone |
| `POST` | `/api/societies/{id}/users/{id}/verify-otp` | Public | Verify OTP and issue JWT |
| `POST` | `/api/auth/password-reset/request` | Public | Request password reset OTP |
| `POST` | `/api/auth/password-reset/confirm` | Public | Confirm OTP and set new password |
| `POST` | `/api/societies/{id}/auth/register` | Public | Self-register via invite link |

---

## Acceptance Criteria
- OTP is delivered via SMS to the registered phone number.
- JWT is not issued until OTP or password is verified.
- Multi-society users must select a society before the JWT is issued.
- `SUAdmin` JWT does not include `apartmentId`.
- `SUUser` JWT includes `apartmentId` for their primary apartment.
- Expired tokens cause the frontend to redirect to login.
- Confirm Password is mandatory on the self-registration form.
- After self-registration, the user lands on the login page, not the password reset page.

## Future / Planned

> 🔜 **Refresh tokens** — issue a long-lived refresh token alongside the access token so users are not logged out on JWT expiry; instead the frontend silently refreshes the access token.

> 🔜 **Session history / login audit** — record login attempts (success and failure) with IP, device, and timestamp for admin audit.

> 🔜 **Social login** — optional sign-in via Google or Microsoft account (commented out in original requirements; deferred).

> 🔜 **MFA / 2FA** — optional second factor beyond OTP for high-privilege accounts such as `HQAdmin` and `SUAdmin`.
